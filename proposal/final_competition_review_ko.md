## 3. 강점

### 1. 분야 3과 매우 잘 맞음

주제가 임상시험 프로토콜, 대상자 적격성, 모집 리스크, 안전성, 데이터 준비도에 집중되어 있다. 이는 분야 3의 "규제 대응 및 지능형 임상 설계"와 직접 연결된다.

### 2. Medical IT 포트폴리오로 설득력 있음

병원정보시스템, EMR/EHR, CRC workflow, 병원 데이터 수집 준비도라는 관점이 살아 있다. 사용자의 의료IT공학 전공 및 병원정보시스템 관련 진로와도 정합성이 높다.

### 3. 실제 MVP가 있음

예선은 제안서 중심이지만, 대회 페이지 설명은 에이전트 AI 기반 신약개발 서비스를 직접 구현하고 산출물을 제출하는 흐름을 강조한다. 현재 프로젝트는 이미 Python CLI, ClinicalTrials.gov API retrieval, scenario run outputs, score sheet를 갖고 있어 단순 아이디어보다 강하다.

### 4. 안전 경계가 좋음

환자별 진단/치료 추천, 실제 환자 데이터 사용, 규제 인증, IRB 승인, 실제 프로토콜 승인 같은 위험한 주장을 피하고 있다. 의료 분야 제안서에서는 이 점이 중요하다.

### 5. 근거 기반 문제정의가 있음

문제정의가 단순 상상이 아니라 ICH, SPIRIT, ClinicalTrials.gov, NCBI, 모집 연구, EHR screening 연구, WHO/FDA guidance와 연결되어 있다.

## 4. 약점 및 리스크

### 1. 분자/화학 중심 주제보다 덜 화려함

분야 1 또는 분야 2의 분자 최적화, 후보물질 발굴, 바이오마커 탐색 주제에 비해 "임상시험 프로토콜 사전검토"는 덜 직접적인 신약개발 AI처럼 보일 수 있다.

대응:

- "후보물질 이후 임상 개발 단계의 실패 리스크를 줄이는 agent"라는 표현을 강화해야 한다.
- 임상시험 설계가 신약개발의 핵심 단계임을 첫 문단에서 더 직접적으로 써야 한다.

### 2. 비즈니스/사회적 가치가 아직 정량적으로 약함

현재 제안은 시간/비용 절감 수치를 과장하지 않아서 안전하다. 하지만 평가표에는 시간·비용 절감 효과가 들어 있으므로 너무 조심하면 약하게 보일 수 있다.

대응:

- "정량 수치는 향후 검증"이라고 두되, 정성적 가치와 측정 계획을 구체화한다.
- 예: 반복 수동 cross-check 감소, registry search trace 자동화, CRC/데이터 담당자 follow-up 질문 생성, protocol review readiness 향상.

### 3. AI 활용 투명성 문구가 더 필요함

공고는 AI 활용 연구 윤리에서 프롬프트, 모델명, 설정값 등 상호작용 과정 기록과 공개를 권고한다.

현재 프로젝트는 GitHub와 산출물 trace가 있지만, 제출 압축본에는 "AI 사용 기록 정책"이 충분히 직접적이지 않다.

대응:

- 제출서 6번 또는 별도 윤리 문단에 다음 문장을 추가한다.

> 본 프로젝트는 AI 활용 과정의 투명성을 위해 주요 프롬프트, 사용 모델, API 질의, 검색 결과, 중간 산출물, 수정 이력을 GitHub와 로컬 산출물로 기록한다. AI가 생성한 문헌·데이터는 실재 여부를 확인한 근거만 제안서에 반영하며, 최종 판단 책임은 연구자에게 둔다.

### 4. Scenario 002/003은 아직 개념 수준

Scenario 001은 강하지만, 추가 평가 시나리오는 아직 구현되지 않았다.

대응:

- 예선 제출에서는 "구현 완료"라고 쓰지 말고 "본선 확장 평가셋"으로 표현한다.
- 본선 진출 시 가장 먼저 Scenario 002/003을 구현한다는 로드맵을 명확히 둔다.

### 5. 최종 HWPX 페이지 검증 필요

DOCX 작업본은 생성되었지만 최종 제출 파일은 HWPX여야 한다. HWPX 변환 후 10페이지 이내인지 확인해야 한다.

대응:

- DOCX에서 먼저 10쪽 이내로 맞춘다.
- 한글에서 HWPX 저장 후 페이지 수를 다시 확인한다.
- 표가 깨지면 표를 문단형으로 줄인다.

## 6. 최종 제출 전 보완 우선순위

### Priority 1: AI 활용 투명성 문단 추가

이유:

- 공고 윤리 가이드라인에 직접 대응한다.
- 짧은 문단 하나로 연구 윤리 및 완성도 점수를 보강할 수 있다.

### Priority 2: "이미 구현된 MVP"를 더 눈에 띄게 표시

이유:

- 예선이 제안서 중심이어도 대회 페이지는 에이전트 서비스 구현을 강조한다.
- 현재 프로젝트의 차별점은 실제 실행 가능한 CLI와 trace 산출물이 있다는 점이다.

권장 문장:

> 현재 MVP는 표준 라이브러리 기반 Python CLI로 구현되어 있으며, 합성 제2형 당뇨병 Phase II 시나리오에 대해 ClinicalTrials.gov 검색, 유사 trial ranking, 병원 데이터 수집 준비도 매핑, 안전성 검토자 루프, 최종 사전검토 보고서 생성을 완료하였다.

### Priority 3: 가치 문단에 측정 계획 추가

이유:

- 시간·비용 절감 수치를 과장하지 않으면서 비즈니스/사회적 가치 점수를 보강할 수 있다.

권장 문장:

> 본선에서는 수작업 검토 대비 agent-assisted review의 비교 지표로 누락 항목 검출 수, 근거 추적 가능 항목 수, follow-up question 품질, 검토자 수정 요청 건수, scenario completion time을 측정한다.

### Priority 4: 최종 제출 HWPX 변환 및 페이지 검수

이유:

- 형식 미준수는 내용이 좋아도 치명적이다.

체크:

- HWPX 파일 형식
- 10쪽 이내
- 분야 3 선택 표시
- 팀명/에이전트명/키워드 입력
- 참고사항 문구 삭제
- 개인정보 공개 여부 확인
