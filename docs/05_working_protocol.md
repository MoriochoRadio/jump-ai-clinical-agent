# Working Protocol

## Why This Exists

This project should be built slowly and carefully. The goal is not to rush into an impressive-looking prototype, but to create a credible Medical IT portfolio project and a competition proposal that can be defended.

## Operating Principles

1. Do not skip problem definition.
2. Do not claim medical, clinical, or regulatory capability without evidence.
3. Prefer narrow, testable workflows over broad all-purpose agents.
4. Document every major decision before implementing it.
5. Keep public portfolio material separate from private submission files.
6. Use public data and public APIs first.
7. Treat AI outputs as draft reasoning that must be checked.
8. Build the smallest useful prototype only after the proposal direction is stable.

## Step Gate

Each phase should have a gate before moving on.

### Gate 1: Competition Understanding

Required before selecting a final idea:

- competition purpose summarized,
- preliminary submission requirements understood,
- evaluation criteria mapped,
- schedule confirmed.

### Gate 2: Career Fit

Required before selecting a final idea:

- connection to Medical IT and hospital information systems is explicit,
- target user is identified,
- workflow is realistic for hospital or clinical research settings.

### Gate 3: Problem Definition

Required before architecture work:

- one specific problem statement,
- why the problem matters,
- current workflow pain points,
- why agentic AI is suitable,
- what the agent should not do.

### Gate 4: Evidence And Tools

Required before prototype work:

- public sources identified,
- tool/API candidates listed,
- risks and limitations documented,
- evaluation scenarios drafted.

### Gate 5: Prototype

Required before building UI:

- target input/output defined,
- agent roles defined,
- evaluation checklist defined,
- minimal happy-path scenario selected.
