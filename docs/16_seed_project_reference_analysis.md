# Seed Project Reference Analysis

## Reference Repository

Reference repository:

- https://github.com/MoriochoRadio/seed-project

The reference project is a completed capstone-style portfolio project for an AI-based sperm analysis system.

Its README uses a polished presentation flow:

1. Introduction
2. Problem
3. How
4. Outcome
5. Artifacts
6. Reference
7. Appendix
8. Quick Start

## What Is Strong About Seed Project

The strongest point is that it starts from a clear problem narrative and then moves toward implementation evidence.

Useful patterns:

- a visible table of contents,
- problem recognition before technical details,
- explicit problem definition,
- actor/user definition,
- architecture diagram,
- outcome and performance section,
- artifact inventory,
- references,
- quick start and project structure.

This makes the project readable as a portfolio, not only as a code repository.

## What Should Be Reused

The current project should reuse the following structural pattern:

```text
Introduction
Problem
How
Outcome
Artifacts
Reference
Appendix
Quick Start
```

For this project, the adapted version should be:

```text
01. Introduction
02. Problem Definition
03. Agent Workflow
04. Current MVP
05. Evidence And Evaluation
06. Artifacts
07. Safety Boundary
08. Next Work
09. Quick Start
```

The most important part to reuse is the problem-first storytelling.

## What Should Not Be Reused Directly

The current project should not copy the completed capstone tone too aggressively.

Reasons:

- `seed-project` is a completed implementation with model training, UI, performance charts, and final deliverables.
- `jump-ai-clinical-agent` is currently a submitted proposal plus early MVP.
- Overstating outcomes would reduce credibility.
- Healthcare AI review support needs cautious language.

In particular, this project should avoid:

- claiming clinical deployment readiness,
- presenting internal scenario results as validated clinical performance,
- using too many badges or visual claims before implementation matures,
- treating the proposal as a completed product.

## Recommended Portfolio Strategy

Use `seed-project` as a reference for sequence, not for tone.

Recommended positioning:

- `seed-project`: completed capstone/product-style portfolio.
- `jump-ai-clinical-agent`: research-backed competition proposal plus reproducible early MVP.

The current repository should emphasize:

- problem definition,
- clinical workflow fit,
- safety boundaries,
- traceable prototype outputs,
- source-backed reasoning,
- next build plan.

## Suggested README Evolution

Short-term README improvements:

- add a compact table of contents,
- add a "Why This Problem" section,
- add a "Current MVP Evidence" section,
- add links to the post-submission retrospective and submission record,
- add one small architecture or workflow diagram if not already visible.

Medium-term README improvements:

- add Scenario 002 results,
- add PubMed retrieval output,
- add a sample generated report screenshot or rendered Markdown preview,
- add a final "Portfolio Notes" section explaining Medical IT relevance.

## Decision

The `seed-project` structure is useful and should be used as a reference.

However, the current project should remain more conservative:

- less product-complete,
- more traceability-focused,
- more explicit about limitations,
- more careful about clinical and regulatory boundaries.

This is the better fit for a submitted healthcare AI proposal and Medical IT portfolio.
